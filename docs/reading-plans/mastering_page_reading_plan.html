Mastering Page Reading Plan

A focused reading order for understanding the John Joseph Mastering microsite, the intake form, the contact flow, and the assets that shape the page.

1. Route and View Entry Points

  1. josephlovesjohn_site/urls.py - confirms that /mastering-services/ is routed into the mastering app.
  2. mastering/urls.py - shows the public page, intake page, and any named routes.
  3. mastering/views.py - contains the landing-page context, contact handling, intake submission, email, and webhook posting.

2. Public Page Template

  1. templates/mastering/home.html - the main microsite page, including inline page-specific styling.
  2. static/mastering/images/ - local optimized images used by the hero, examples, and lower service cards.
  3. static/assets/js/main.js - bundled HTML5 UP shell behavior used by the Solid State base.
  4. static/assets/css/main.css - the inherited Solid State theme base before page-specific overrides.

3. Intake Form Flow

  1. mastering/forms.py - role-specific artist and mix engineer questions, validation, and cleaned values.
  2. templates/mastering/intake.html - frontend role switching, field grouping, date input styling, and checkbox/radio presentation.
  3. mastering/views.py - the intake POST flow, email body, Google Sheets payload, and success/error handling.
  4. josephlovesjohn_site/settings.py - environment variables for email recipients and MASTERING_INTAKE_GOOGLE_SHEETS_WEBHOOK_URL.

4. Contact and External Services

  1. main_site/forms.py - shared contact form patterns if comparing the main-site and mastering contact behavior.
  2. mastering/views.py - mastering contact submission and recipient selection.
  3. josephlovesjohn_site/csp.py - allowed origins for embeds, static assets, analytics, and third-party content.
  4. .env.example - local configuration names without exposing private credentials.

5. Tests That Explain Expected Behavior

  1. tests/test_mastering_smoke.py - page rendering, key copy, examples, sections, and template expectations.
  2. tests/test_mastering_routes.py - URL availability and route behavior.
  3. tests/test_mastering_contact_form.py - mastering contact flow behavior.
  4. tests/test_mastering_intake_form.py - role-specific intake validation and payload expectations.

Questions To Keep In Mind

  • Is this part of the public sales page, the private intake workflow, or shared site infrastructure?
  • Which content should stay static in the template and which should become configurable later?
  • Does this change affect email delivery, Google Sheets payloads, or only presentation?
  • Which test would fail if this behavior accidentally changed?
