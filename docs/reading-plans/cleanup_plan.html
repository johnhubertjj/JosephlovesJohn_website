Repository Cleanup Plan

A practical cleanup plan for the JosephlovesJohn website codebase, focused on production safety, maintainability, and clarity rather than large-scale rewrites.

Do Now

  • Enforce production config safety in josephlovesjohn_site/settings.py, especially around SECRET_KEY and ALLOWED_HOSTS.
  • Split the largest coordination files before they get harder to change safely: shop/views.py, main_site/site_data.py, and static/main_site/js/site.js.
  • Replace dict-shaped gallery payloads with typed structures such as TypedDict or dataclass-style objects in the art/media builders.
  • Keep repository metadata aligned with actual releases, runtime versions, and deployment expectations.

Do Soon

  • Extract shop service logic from view functions into smaller modules for checkout fulfillment, download authorization, and account/order helpers.
  • Split main-site asset and content builders into narrower modules such as gallery assets, navigation data, and music library data.
  • Break the site shell JavaScript into modules by concern: routing, lightbox/media, signup/embed, and analytics/cookies.
  • Add a short internal architecture note describing request flow, shared content caching, and public asset resolution.

Nice Later

  • Add a lightweight health/readiness endpoint for deploy verification.
  • Move more side effects off the request path if traffic grows.
  • Standardize private helper naming and reduce cross-module imports of underscored functions.

Guiding Principle

The codebase already works well enough for production. The goal is not to rewrite it into a framework exercise. The best next improvements are the ones that reduce maintenance drag, sharpen production safety, and make future changes easier to reason about.
