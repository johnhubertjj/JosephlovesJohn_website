# JosephlovesJohn Website

This repository contains the Django code for the JosephlovesJohn website.

It includes:

- the main artist site
- the digital-download shop
- the mastering microsite
- legal pages, contact form, cart, checkout, and customer download flows

## Stack

- Python 3.14
- Django 5
- JavaScript for the site shell, media players, cart UI, cookie controls, and interactive embeds
- Stripe Checkout for payments
- Kit (ConvertKit) for mailing-list signup embeds
- WhiteNoise for static file serving
- Gunicorn as the production application server
- Render for deployment
- Cloudflare R2 for large public and private media assets
- PostgreSQL in production and SQLite locally by default
- `uv` for dependency management
- Pytest for the test suite
- Playwright for browser/UI coverage
- Ruff for linting
- mypy with `django-stubs` for type checking
- HTML5 UP templates as the visual base, including the Solid State starter
- jQuery and the bundled HTML5 UP frontend utilities
- Font Awesome for iconography
- PlayerJS for the music player embed layer
- Sentry support for error monitoring

## Local Development

### 1. Install dependencies

```bash
uv sync --dev
```

### 2. Create a local environment file

Copy [`.env.example`](/Users/johnjoseph/PycharmProjects/JosephlovesJohn_website/.env.example) to `.env` and fill in the values you need.

This project now auto-loads `.env` in local development from the repo root, so you do not need a separate dotenv tool just to run the site locally.

### 3. Run migrations

```bash
uv run python manage.py migrate
```

### 4. Create an admin user

```bash
uv run python manage.py createsuperuser
```

### 5. Start the dev server

```bash
uv run python manage.py runserver
```

The site will then be available at [http://127.0.0.1:8000](http://127.0.0.1:8000).

## Environment Notes

### Local development

- SQLite is used by default when `DATABASE_URL` is unset.
- `.env` is intended for local development.
- Console email is a good default locally.

### Production

- Render should use `DATABASE_URL` for Postgres.
- Large media files should live in Cloudflare R2 rather than Git.
- [`.env-render-only`](/Users/johnjoseph/PycharmProjects/JosephlovesJohn_website/.env-render-only) is a private reference file for Render settings.

## Media and Asset Storage

This project supports a split storage model:

- repo-tracked static assets for CSS, JS, icons, and smaller site assets
- public Cloudflare R2 assets for artwork, gig photos, and optionally preview audio
- private Cloudflare R2 assets for paid downloads

Keep the same relative keys when uploading to R2. For example:

- local file: `static/audio/example.mp3`
- R2 key: `audio/example.mp3`

- local file: `static/images/album_art/cover.jpg`
- R2 key: `images/album_art/cover.jpg`

Do not upload `staticfiles/` to R2. That directory is generated by `collectstatic`.

See:

- [Cloudflare R2 setup](/Users/johnjoseph/PycharmProjects/JosephlovesJohn_website/docs/cloudflare_r2.md)
- [Deployment notes](/Users/johnjoseph/PycharmProjects/JosephlovesJohn_website/docs/deployment.md)

```bash

```

Note: the repo contains older migration files that may still trigger line-length warnings if you run Ruff against everything. In practice, we usually lint the active Python modules we touched.

## Deployment

Recommended production shape:

- 1 Render web service
- 1 Render Postgres database
- optional Render persistent disk for `media/` while migrating
- Cloudflare R2 for large public/private assets

## Payments

- Checkout is handled with Stripe-hosted Checkout pages.
- Order records and download entitlements are stored in Django.
- Paid downloads can be served from a private R2 bucket using short-lived signed URLs.

## Browser/UI Notes

- The main site is a single-page shell with route-aware section activation.
- Legal pages are standalone routed pages.
- The mailing-list signup form now loads only after user interaction unless optional cookies have already been allowed.

## Status

This repo is actively used for the live JosephlovesJohn website, so it is best treated as an application codebase rather than a generic starter template.
